# 架构任务拆分

把设计方案转化为没有隐含上下文、边界明确、依赖可执行的任务文档。必须实际读取相关文档和代码；不得仅根据用户摘要生成通用清单。

## 默认交付物

若用户未另行指定：

```text
docs/tasks/
├── setup.md
├── task-001-<slug>.md
├── task-002-<slug>.md
└── ...
```

已有文件时先检查其用途，保留仍然正确的内容。不要覆盖无关文档。

## 工作流程

### 1. 建立事实基础

1. 阅读用户指定的全部架构、需求和约束文档。
2. 检查与方案直接相关的代码、配置、测试、构建入口和目录约定。
3. 区分：
   - **已确认事实**：文档或代码直接支持；
   - **合理推导**：由现有接口或约束推得，需写明依据；
   - **未决事项**：缺少依据且会影响实现。
4. 优先通过代码和文档消除不确定性。只有会实质改变拆分结果且无法自行判断时才询问用户。
5. 不得虚构不存在的模块、接口、命令、技术栈或目录。计划新增的内容明确标注为“新增”。

### 2. 确定实施边界

从目标反推端到端能力，至少覆盖适用的方面：

- 领域模型、数据结构与状态转换；
- 后端服务、API、异步作业或 Agent 执行链路；
- 存储、迁移、幂等性与一致性；
- 前端或调用方集成；
- 权限、安全、隔离与敏感数据；
- 可观测性、失败恢复与运维；
- 单元、集成、端到端测试；
- 配置、开发环境、文档和发布迁移。

明确本次范围内与范围外内容。架构文档存在内部矛盾时，先指出并尽量依据代码和上下文解决；无法解决的矛盾作为阻塞项，不得静默选择。

### 3. 拆分任务

每个任务必须：

- 交付一个可描述、可验证的工程结果，而不是“研究”“处理”“完善”等开放动作；
- 边界内聚，避免两个任务修改同一职责；
- 列出输入、输出、涉及组件及明确的不做事项；
- 指明新增或修改的文件/目录；若路径尚无法确定，给出定位规则，不能捏造路径；
- 明确接口、数据契约、错误语义和关键边界条件；
- 包含可操作的验证步骤与成功判据；
- 具备依赖任务完成后即可执行的全部上下文；
- 工作量适合一次实现和一次审查。若横跨多个独立组件或无法独立验证，继续拆分。

共享契约应由最早的基础任务定义，后续任务引用该任务和 `setup.md`，不要在不同任务中重复定义相互冲突的版本。

### 4. 编排依赖与并行工作

1. 为任务分配稳定 ID：`TASK-001`、`TASK-002`……
2. 依赖只能引用已定义的任务 ID，依赖图必须无环。
3. 区分：
   - **硬依赖**：前置任务的产物是当前任务的必要输入；
   - **协作关系**：可并行，但需共同遵守接口契约或最终联调。
4. 给出执行波次：
   - 同一波次内的任务可并行；
   - 后续波次只依赖已完成波次；
   - 最终集成任务不得代替各任务自身的验证。
5. 尽量减少共享文件冲突。无法避免时，指定所有者、合并顺序或稳定扩展点。

## `setup.md` 必备结构

`setup.md` 是所有执行者共享的唯一入口，必须包含：

```markdown
# 实施上下文

## 1. 项目与目标
- 当前系统做什么
- 本次要实现的用户/业务结果
- 明确的成功标准

## 2. 事实来源
- 架构/需求文档及各自作用
- 相关代码入口、配置和测试位置
- 文档与现状的差异

## 3. 当前架构
- 组件职责与调用/数据流
- 关键领域对象和状态
- 已有约束与不可破坏的不变量

## 4. 目标架构与统一约定
- 目标流程
- 跨任务共享的接口、模型、命名和错误约定
- 安全、可观测性、兼容性要求

## 5. 实施范围
### 范围内
### 范围外

## 6. 任务总览
- 每个任务的 ID、标题、交付结果、硬依赖、可并行关系

## 7. 依赖图与执行波次
- Mermaid DAG
- 按波次列出并行安排

## 8. 全局验收标准
- 端到端行为
- 质量门槛
- 回归要求

## 9. 未决事项与假设
- 问题、影响、当前处理方式、需要谁在何时确认
```

`setup.md` 应提供完成任意任务所需的全局背景，但不要复制整份原始架构文档。用精确摘要和文件引用保留可追溯性。
文件引用仅用于追溯，不能代替执行上下文；实现者不得被要求再阅读原始架构文档才能完成任务。

## 每个任务文档的必备结构

```markdown
# TASK-NNN：<可交付结果>

## 元数据
- 状态：待实施
- 执行波次：<N>
- 硬依赖：<任务 ID 或“无”>
- 协作关系：<任务 ID、共享契约/冲突点或“无”>
- 预计改动范围：<组件和文件/目录>

## 目标
<完成后可观察到的结果及其价值>

## 上下文
<包含该任务特有背景；引用 setup.md 的统一约定。所有实施必需信息必须写在这两个文件中，其他来源仅作追溯>

## 前置条件
<依赖任务必须提供的确切产物、环境和决策>

## 实施范围
### 必须完成
### 明确不包含

## 设计与执行细节
1. <按可执行顺序写步骤>
2. <指定数据、接口、控制流和错误处理>
3. <列出兼容性、并发、幂等或安全边界>

## 接口与数据契约
<请求、响应、事件、模型、配置或“不涉及”；说明字段语义和失败行为>

## 文件变更
- `path`：新增/修改，以及具体职责

## 验证
### 自动化验证
- <具体测试层级、场景、命令（仅使用已确认存在的命令）和预期结果>
### 手工验证
- <必要时给出步骤和可观察结果；不需要则写“不需要”>
### 边界与失败场景
- <至少覆盖与本任务相关的关键边界>

## 完成定义
- [ ] <可客观判定的实现结果>
- [ ] <测试和回归结果>
- [ ] <文档、配置或可观测性要求>

## 风险与注意事项
<具体风险、缓解方式；无则写“无已知特有风险”>
```

不要用“按最佳实践实现”“补充必要测试”“处理异常”“视情况修改”等表述代替决策。应写出适用的实践、测试场景、异常类型、触发条件和预期行为。

## 一致性与可执行性审查

初稿完成后必须逐项审查并直接修复发现的问题：

1. **覆盖性**：架构中的每个范围内能力都映射到至少一个任务；没有无人负责的集成、迁移或测试。
2. **边界**：任务无职责重叠；同一产物只有一个明确所有者。
3. **依赖**：所有 ID 有效、无环，执行波次与硬依赖一致；并行任务不依赖彼此未完成的实现。
4. **契约**：`setup.md` 与任务中的接口、模型、命名、路径和错误语义一致。
5. **可执行性**：假设执行者只读取 `setup.md` 和当前任务文档，仍能判断改哪里、如何实现、何时完成。
6. **可验证性**：每项完成定义均有对应验证；验证包含正常、关键边界和失败路径。
7. **真实性**：路径、命令、现有能力均有来源；计划新增内容已明确标注。
8. **防漂移**：每个任务都有“不包含”项，未决问题没有被伪装成实现细节。

审查后重新读取所有输出文件做交叉检查。只在上述问题已修复后报告完成；最终说明输出位置、任务数量、并行波次及仍需用户决策的阻塞项。
