# 游戏音频引擎设计技术文档

## 1. 设计哲学与目标

### 1.1 生产导向设计原则

本音频引擎的设计核心原则是**从实际游戏生产需求出发，而非从现有引擎方案照搬**。与 Wwise 和 FMOD 等商业中间件不同，本引擎不追求功能列表的完整性，而是追求在真实项目约束下的技术决策合理性。设计过程中遵循以下铁律：

- **成本前置消灭**：所有性能开销在编译期或资产预处理期即被量化，运行时不允许出现不可预测的 CPU 尖峰。这与用户现有 RenderScript DSL 的设计理念一脉相承——将不确定性从热路径中驱逐出去。
- **数据驱动优先于代码驱动**：音频行为（衰减曲线、混响参数、优先级策略）全部由数据资产定义，而非硬编码在 C/C++ 逻辑中。这确保了音频设计师可以在不触碰引擎代码的前提下完成迭代。
- **底层原理透明**：不将任何 DSP 算法视为"黑盒"。引擎开发者必须能够追踪从采样点到扬声器振膜的完整信号链，理解每一步的数学变换。这符合用户"只有弄清楚原理，才能掌握本质"的技术信念。

### 1.2 与现有引擎架构的集成策略

根据用户的画像，其正在设计的渲染架构采用 **L0-L9 十层架构**，底层基于 **slang + slang-rhi** 实现通用 RHI 层，对外暴露 C API，支持多线程渲染、Bindless、Indirect Draw、Async Compute 等现代特性 [用户画像]。音频引擎将遵循相同的分层哲学，与渲染管线共享以下基础设施：

| 共享组件 | 渲染侧用途 | 音频侧用途 |
|:---|:---|:---|
| **通用 VM** | RenderScript DSL 执行 | 音频效果器脚本热更新 |
| **RHI 抽象层** | GPU 资源管理 | 硬件加速音频接口封装 |
| **多线程调度器** | Async Compute 调度 | 音频线程与 Worker 线程池 |
| **资产管线** | 纹理/网格编译 | 音频 Bank 编译与元数据生成 |

音频引擎不依赖渲染侧的 GPU 资源，但通过统一的 C API 层与渲染引擎共享线程模型、内存分配器和日志系统。这种**共生而非寄生**的集成策略避免了跨引擎的 FFI 开销，同时保持了模块间的清晰边界。

## 2. 分层架构设计

音频引擎采用与用户现有架构一致的十层分层模型，从硬件抽象到高层语义逐级构建。每一层只依赖其下层，不跨层调用。

### 2.1 架构总览

```
┌─────────────────────────────────────────────────────────────┐
│  L9: 高层接口层 — Event System, Script Binding, Editor API │
├─────────────────────────────────────────────────────────────┤
│  L8: 资源管理层 — Voice Lifecycle, Audio Bank, Streaming   │
├─────────────────────────────────────────────────────────────┤
│  L7: 空间音频层 — HRTF, Ambisonics, Propagation Model     │
├─────────────────────────────────────────────────────────────┤
│  L6: 效果器层 — FDN Reverb, Dynamics, EQ, Filter Chain    │
├─────────────────────────────────────────────────────────────┤
│  L5: 混音核心层 — Mixing Engine, Bus Routing, Voice Mgmt  │
├─────────────────────────────────────────────────────────────┤
│  L4: DSP 原语层 — Biquad, FFT, Convolution, Oscillator    │
├─────────────────────────────────────────────────────────────┤
│  L3: 采样转换层 — SRC, Format Conversion, Dither           │
├─────────────────────────────────────────────────────────────┤

```

### 2.2 L0-L2：平台抽象层

#### 2.2.1 L0：操作系统接口

L0 层直接封装操作系统原语，不引入任何抽象。核心职责包括：

- **内存管理**：提供预分配内存池（Arena Allocator），禁止在音频热路径中调用 `malloc`/`free`。所有动态分配必须在初始化阶段完成。
- **线程模型**：创建高优先级实时线程（`SCHED_FIFO` on Linux, `AVAudioEngine` 的实时线程 on Apple），与主线程通过无锁队列通信。
- **SIMD 检测**：运行时检测 CPU 的 SIMD 能力（SSE/AVX/NEON），选择最优的向量化路径。

#### 2.2.2 L1：硬件驱动层

L1 层负责设备枚举与流管理。核心数据结构为 `DeviceHandle`，封装了跨平台的设备标识符：

- **Windows**：通过 WASAPI 的 `IMMDeviceEnumerator` 枚举设备，使用 `IAudioClient3` 获取低延迟独占模式 [9]。
- **macOS/iOS**：通过 Core Audio 的 `AudioObjectGetPropertyData` 枚举设备，使用 `kAudioHardwarePropertyDevices` 获取设备列表。
- **Android**：通过 AAudio 的 `AAudio_createStreamBuilder` 构建流，优先使用 MMAP 模式以降低延迟 [9]。

#### 2.2.3 L2：平台抽象层

L2 层提供统一的 `AudioDevice` API，屏蔽底层差异。核心接口设计如下（Zig 风格伪代码）：

```zig
// L2 平台抽象层核心接口
pub const AudioDevice = struct {
    // 设备能力描述
    capabilities: DeviceCapabilities,
    // 当前流配置
    stream_config: StreamConfig,

    // 打开设备并启动音频流
    pub fn open(device_id: []const u8, config: StreamConfig) !AudioDevice;
    // 关闭设备，释放所有资源
    pub fn close(self: *AudioDevice) void;
    // 获取下一帧音频数据的写入指针（零拷贝）
    pub fn getBuffer(self: *AudioDevice) ![]f32;
    // 提交已填充的音频帧
    pub fn commitBuffer(self: *AudioDevice, frames: u32) !void;
};

pub const StreamConfig = struct {
    sample_rate: u32,       // 典型值：48000
    channels: u16,           // 典型值：2（立体声）或 6（5.1）
    buffer_frames: u16,     // 典型值：256（约 5.3ms @48kHz）
    format: SampleFormat,    // f32 或 i16
};

pub const SampleFormat = enum {
    f32,
    i16,
    i24_packed,
};
```

跨平台延迟预算参考：

| 平台 | API | 典型往返延迟 | 独占模式 | 备注 |
|:---|:---|:---|:---|:---|
| Windows | WASAPI | 2-10ms | 支持 | 绕过系统混音器 [9] |
| macOS/iOS | Core Audio | 5-10ms | 默认 | 架构原生低延迟 |
| Android | AAudio (MMAP) | 3-20ms | 不支持 | 8.0+ 设备支持 [9] |
| Linux | ALSA/PipeWire | 2-15ms | 部分支持 | 依赖发行版配置 |

### 2.3 L3-L4：核心 DSP 层

#### 2.3.1 L3：采样转换层

L3 层负责音频格式的标准化转换。所有进入混音引擎的音频数据必须统一为 **48kHz / 32-bit 浮点 / 单声道** 的内部格式。这一决策基于以下工程考量：

- **48kHz 采样率**：满足 Nyquist 定理对 20kHz 人耳上限的要求，同时与主流游戏音频中间件（Wwise、FMOD）的内部采样率一致，避免不必要的 SRC 开销。
- **32-bit 浮点**：提供 1528dB 的理论动态范围，确保混音过程中不会发生内部削波。最终输出时通过 TPDF 抖动降至 16-bit 或 24-bit 整数 [10]。
- **单声道**：3D 空间化要求音源必须为单声道，HRTF 处理将单声道渲染为双耳立体声 [7]。

采样率转换（SRC）采用 **sinc 插值 + 线性插值** 的分级策略：

- **高质量 SRC**：使用 64-tap Kaiser 窗 sinc 函数，阻带衰减 >100dB，用于资产预处理阶段。
- **低延迟 SRC**：使用 8-tap 线性插值，用于运行时实时变调（多普勒效应）。

#### 2.3.2 L4：DSP 原语层

L4 层提供所有 DSP 算法的原子实现，是效果器链的构建基础。核心原语包括：

**Biquad 滤波器**：实现 Direct Form II Transposed 结构，因其在浮点运算下的数值稳定性优于 Direct Form I [11]。差分方程为：

$$y[n] = b_0 x[n] + b_1 x[n-1] + b_2 x[n-2] - a_1 y[n-1] - a_2 y[n-2]$$

系数计算遵循 RBJ Cookbook 的标准化公式，支持以下滤波器类型：

| 滤波器类型 | 参数 | 典型应用 |
|:---|:---|:---|
| Lowpass | 截止频率, Q | 距离衰减的高频吸收 |
| Highpass | 截止频率, Q | 风声去除、隆隆声消除 |
| Peaking | 中心频率, 增益, Q | 脚步声材质共振 |
| Low Shelf | 截止频率, 增益 | 水下音效的低频增强 |
| High Shelf | 截止频率, 增益 | 电话音效的高频衰减 |

**FFT 卷积引擎**：基于 Split-Radix FFT 实现，支持 Overlap-Add 和 Overlap-Save 两种分块策略。对于长脉冲响应（>8192 样本），自动切换为分区卷积（Partitioned Convolution），将 IR 切分为多个子块以平衡延迟与吞吐量 [1]。

**振荡器**：提供正弦、锯齿、方波、三角波四种基础波形，以及基于波表（Wavetable）的任意波形合成。所有振荡器支持相位同步（Phase Sync），避免多音叠加时的相位抵消。

### 2.4 L5-L6：混音与效果器层

#### 2.4.1 L5：混音核心层

L5 是音频引擎的**心脏**——所有活跃语音在此汇聚、混合并输出。核心设计围绕以下三个数据结构展开：

**Audio Bus（音频总线）**：有向无环图（DAG）结构，每个 Bus 节点包含：
- 输入缓冲区（环形缓冲，256 样本粒度）
- 增益控制（dB 标度，支持 Automation）
- 效果器插入点（Pre-Fader / Post-Fader）
- 子 Bus 列表（支持层级路由）

**Voice（语音）**：代表一个正在播放的音频实例，包含：
- 当前播放位置（样本级精度）
- 音量包络（Attack/Decay/Sustain/Release）
- 优先级（0-255，0 为最高优先级）
- 所属 Bus 路由
- Virtual Voice 标记位

**Voice Stealing 算法**：当活跃语音数超过上限（典型值 64-128）时，触发语音窃取。采用三级决策：

1. **优先级最低优先**：按优先级排序，淘汰最低优先级的语音。
2. **音量最静优先**：在同等优先级下，淘汰当前 RMS 最低的语音。
3. **距离最远优先**：在同等优先级和音量下，淘汰距离 Listener 最远的语音。

这一策略参考了 Wwise 的 Virtual Voice 机制，但增加了距离维度的考量，更适合开放世界场景 [10]。

**混音循环伪代码**（Zig 风格）：

```zig
// L5 混音核心循环 - 每帧执行一次
pub fn mixCycle(mixer: *AudioMixer, output_buffer: []f32) void {
    // 1. 遍历所有活跃 Voice，执行 Voice Stealing 判定
    mixer.voice_manager.stealIfNeeded();

    // 2. 对每个活跃 Voice，拉取下一帧音频数据
    for (mixer.voice_manager.active_voices) |*voice| {
        if (voice.is_virtual) continue;  // 虚拟语音跳过 DSP

        // 从环形缓冲区读取 256 样本
        const input = voice.ring_buffer.read(FRAMES_PER_BUFFER);

        // 3. 应用音量包络和距离衰减
        const gain = voice.envelope.tick() * voice.distance_attenuation;

        // 4. 将处理后的样本累加到目标 Bus
        const bus = mixer.bus_graph.resolve(voice.bus_id);
        bus.accumulate(input, gain);
    }

    // 5. 遍历 Bus 图，执行效果器链
    mixer.bus_graph.processEffects();

    // 6. 最终混音到输出缓冲区
    mixer.bus_graph.mixdown(output_buffer);
}
```

#### 2.4.2 L6：效果器层

L6 层在 L4 原语基础上构建完整的效果器链。每个效果器必须实现 `EffectProcessor` 接口：

```zig
pub const EffectProcessor = struct {
    // 处理 256 样本的音频块
    processFn: *const fn (self: *anyopaque, input: []const f32, output: []f32) void,
    // 重置效果器状态（用于语音回收）
    resetFn: *const fn (self: *anyopaque) void,
    // 效果器参数更新（线程安全）
    setParameterFn: *const fn (self: *anyopaque, param_id: u32, value: f32) void,
};
```

**FDN 混响**：采用 16 路延迟线 + Householder 反馈矩阵的 FDN 结构，回声密度呈指数增长，音质自然无金属感 [9]。反馈矩阵 $A$ 满足 $A^T A = I$（正交性），确保系统能量守恒。核心参数：

- **RT60**：混响时间，控制延迟线的衰减系数
- **Pre-Delay**：早期反射的预延迟时间（0-100ms）
- **ER/LR Ratio**：早期反射与后期混响的能量比
- **Diffusion**：扩散系数，控制回声密度的增长速度

**动态压缩器**：采用前馈拓扑结构，通过侧链检测信号包络。增益计算公式为 [9]：

$$G = (Threshold - Input) \times \left(1 - \frac{1}{Ratio}\right)$$

其中 $Input$ 为侧链检测到的 RMS 电平（dB），$Threshold$ 为压缩阈值（dB），$Ratio$ 为压缩比（如 4:1）。Attack（5-50ms）和 Release（20-200ms）时间常数通过一阶 IIR 平滑滤波器实现。

### 2.5 L7-L8：空间音频与资源管理

#### 2.5.1 L7：空间音频层

L7 层负责将单声道音源渲染为双耳立体声或环绕声输出。核心子系统包括：

**HRTF 处理管线**：

1. **HRTF 数据集加载**：从资产包加载预处理的 HRIR（Head-Related Impulse Response）数据集。支持 CIPIC、MIT KEMAR 等标准数据集，以及通过 AI 生成的个性化 HRTF Profile [3]。
2. **方位角插值**：对 Listener-Source 的相对方位角（Azimuth）和仰角（Elevation）进行双线性插值，从最近的 4 个 HRIR 采样点合成当前方向的脉冲响应。
3. **时域卷积**：将插值后的 HRIR 与单声道音源进行 128-tap 或 256-tap 的直接卷积（Direct Convolution），复杂度 $O(N)$，无需 FFT。

**距离衰减模型**：遵循反平方律，距离每增加一倍，声压级下降 6dB。实际实现中引入可配置的衰减曲线：

$$A(d) = \frac{1}{\left(\frac{d}{d_{ref}}\right)^\alpha}$$

其中 $d_{ref}$ 为参考距离（通常为 1m），$\alpha$ 为衰减指数（自由场 = 2，游戏内通常为 1.5-2.5）。为防止近场爆炸，在 $d < d_{min}$ 时钳制增益。

**多普勒效应**：实时计算相对速度引起的频移 [10]：

$$f_{observed} = f_{source} \times \frac{c + v_{listener}}{c + v_{source}}$$

其中 $c$ 为声速（343 m/s），$v_{listener}$ 和 $v_{source}$ 分别为听者和声源沿视线方向的速度分量。实现时通过 L3 层的低延迟 SRC 完成实时变调。

**遮挡与衍射**：基于声学射线追踪（Acoustic Ray Tracing）的简化模型。从声源向 Listener 发射多条射线，统计被几何体遮挡的射线比例，计算遮挡系数：

$$Occlusion = 1 - \frac{N_{unoccluded}}{N_{total}}$$

衍射效应通过几何绕射理论（GTD）的简化模型处理，在遮挡边缘引入额外的频率相关衰减。这一设计预留了与 Xbox Project Acoustics 类似硬件加速接口的集成空间 [12]。

**Ambisonics 编解码**：支持一阶（First-Order, 4 通道）到三阶（Third-Order, 16 通道）的 Ambisonics 编码。通道数遵循 $(N+1)^2$ 公式 [9]。解码器使用虚拟扬声器阵列（Virtual Speaker Array），将 B-Format 信号渲染至任意输出布局。

#### 2.5.2 L8：资源管理层

L8 层管理音频资产的生命周期，是连接资产管线与运行时混音引擎的桥梁。

**Audio Bank 格式**：音频资产以 Bank 为单位组织，每个 Bank 包含：

```zig
// L8 音频资产包格式 - 编译期生成
pub const AudioBank = struct {
    // 文件头
    magic: u32,                    // 0x41454442 ("AEDB")
    version: u16,                  // 格式版本号
    flags: u16,                    // 平台标志位

    // 元数据区
    sounds: []SoundEntry,          // 音效条目数组
    buses: []BusEntry,            // Bus 配置数组
    effects: []EffectPreset,      // 效果器预设数组

    // 数据区
    pcm_data: []u8,              // PCM 音频数据（可压缩）
    hrir_data: []u8,             // HRIR 脉冲响应数据
    ir_data: []u8,                // 混响脉冲响应数据
};

pub const SoundEntry = struct {
    id: u32,                      // 运行时引用 ID
    name: []const u8,             // 编辑器显示名称
    format: AudioFormat,          // 编码格式
    sample_rate: u32,             // 原始采样率
    channels: u8,                 // 原始声道数
    data_offset: u32,            // 在 pcm_data 中的偏移
    data_size: u32,              // 数据长度（字节）
    loop_points: ?[2]u32,       // 循环点（可选）
};
```

**流式加载策略**：根据音频类型采用分级加载策略：

| 音频类型 | 加载方式 | 内存占用 | 延迟要求 |
|:---|:---|:---|:---|
| 枪声/UI 音效 | 全量预加载 | 完整文件 | <1ms |
| 脚步声/Foley | 首帧 + 流式 | 256 样本缓冲 | <5ms |
| 背景音乐 | 流式加载 | 2x 环形缓冲 | <50ms |
| 环境长音 | 流式加载 | 4x 环形缓冲 | <100ms |

**Voice 生命周期管理**：Voice 对象池采用预分配策略，避免运行时分配。每个 Voice 的状态机如下：

```
[Free] → [Requested] → [Active] → [Virtual] → [Stolen] → [Free]
                          ↓
                       [Releasing] → [Free]
```

Virtual Voice 是关键的 CPU 优化手段：当语音的音量低于可听阈值（-64dB）时，Voice 进入 Virtual 状态，仅跟踪播放进度而不执行 DSP 处理。当音量回升时，Voice 无缝切换回 Active 状态 [10]。

### 2.6 L9：高层接口层

L9 层面向游戏逻辑和音频设计师提供语义化接口，隐藏底层 DSP 复杂性。

**事件系统**：采用事件驱动模型，游戏代码通过 `AudioEvent` 触发声音，而非直接操作 Voice 对象：

```zig
pub const AudioEvent = struct {
    event_type: AudioEventType,
    sound_id: u32,
    position: ?Vec3,           // 3D 空间位置（可选）
    parent_id: ?u32,          // 附着对象 ID（可选）
    parameters: []Parameter,   // 运行时参数覆盖
};

pub const AudioEventType = enum {
    play_one_shot,             // 一次性播放
    play_looping,             // 循环播放
    stop,                     // 停止
    pause,                   // 暂停
    resume,                  // 恢复
    set_parameter,           // 参数更新
};
```

## 3. 底层技术原理深入

### 3.1 数学基础

#### 3.1.1 采样定理与量化噪声

Nyquist-Shannon 采样定理是数字音频的基石：为避免混叠，采样频率 $f_s$ 必须严格大于信号最高频率的两倍 [10]：

$$f_s > 2f_{max}$$

在 48kHz 采样率下，可完美重建 24kHz 以下的信号，覆盖人耳听觉范围（20Hz-20kHz）并留有足够的过渡带供抗混叠滤波器滚降。

量化噪声的信噪比（SNR）由位深 $n$ 决定 [10]：

$$SNR \approx 6.02n + 1.76 \text{ dB}$$

| 位深 | 理论 SNR | 动态范围 | 典型应用 |
|:---|:---|:---|:---|
| 8-bit | 49.9 dB | 48 dB | 复古音效、Lo-Fi 效果 |
| 16-bit | 98.1 dB | 96 dB | CD 音质、最终输出 |
| 24-bit | 146.2 dB | 144 dB | 录音棚、内部处理 |
| 32-bit float | 1528 dB | 实际无限 | 内部混音总线 |

TPDF 抖动的概率密度函数为两个均匀分布的卷积，呈三角形状。其数学性质保证了一阶矩（均值）和二阶矩（方差）的噪声特性与输入信号无关，完全消除量化失真产生的谐波 [10]。

#### 3.1.2 球谐函数与 Ambisonics

Ambisonics 的空间编码基于球谐函数（Spherical Harmonics）$Y_{nm}(\theta, \phi)$，其中 $n$ 为阶数，$m$ 为模式。一阶 Ambisonics 使用 4 个球谐基函数编码声场：

$$Y_{00}(\theta, \phi) = \frac{1}{\sqrt{4\pi}} \quad \text{(全向, W)}$$
$$Y_{1-1}(\theta, \phi) = \sqrt{\frac{3}{4\pi}} \sin(\phi) \sin(\theta) \quad \text{(前后, Y)}$$
$$Y_{10}(\theta, \phi) = \sqrt{\frac{3}{4\pi}} \cos(\theta) \quad \text{(上下, Z)}$$
$$Y_{11}(\theta, \phi) = \sqrt{\frac{3}{4\pi}} \sin(\phi) \cos(\theta) \quad \text{(左右, X)}$$

第 $N$ 阶 Ambisonics 的通道数为 $(N+1)^2$，空间分辨率随阶数提升而指数级提高 [9]。

#### 3.1.3 FFT 卷积定理

卷积定理将时域卷积转化为频域乘法，是 FDN 混响和 HRTF 卷积的数学基础 [1]：

$$\mathcal{F}\{x * h\} = \mathcal{F}\{x\} \cdot \mathcal{F}\{h\}$$

对于长度为 $N$ 的信号和长度为 $M$ 的脉冲响应，直接卷积复杂度为 $O(NM)$，FFT 卷积复杂度为 $O((N+M) \log (N+M))$。当 $M > 64$ 时，频域方法开始显现优势。

### 3.2 系统层原理

#### 3.2.1 实时线程约束

音频线程必须满足**实时安全（Real-time Safe）** 约束。违反此约束的任何操作（如内存分配、文件 I/O、锁竞争）都可能导致音频丢帧（Dropout）或爆音（Glitch）[8]。

实时安全的核心规则：

1. **禁止动态内存分配**：`malloc`/`free` 的延迟不可预测（可能触发页错误或系统调用）。所有内存必须在初始化阶段预分配。
2. **禁止系统调用**：`fopen`/`fread`/`fwrite` 涉及磁盘 I/O 和内核态切换，延迟可达毫秒级。文件读取必须在非实时线程完成。
3. **禁止锁竞争**：`mutex.lock()` 可能导致优先级反转。线程间通信必须使用无锁数据结构（Lock-Free Queue）。
4. **禁止阻塞等待**：任何形式的 `sleep`、`wait` 或 `yield` 都会破坏音频线程的周期性调度。

#### 3.2.2 无锁编程与 SPSC 队列

音频引擎使用 **SPSC（Single Producer Single Consumer）** 环形缓冲区作为线程间通信的基础设施。SPSC 队列的数学性质保证了在单一生产者和单一消费者的约束下，无需任何原子操作即可实现线程安全 [8]。

```zig
// SPSC 环形缓冲区 - 无锁实现
pub const RingBuffer = struct {
    buffer: []f32,
    capacity: u32,
    // 读写指针：仅由各自线程修改，无需原子操作
    read_pos: u32 align(64),   // Cache Line 隔离
    write_pos: u32 align(64),  // Cache Line 隔离

    pub fn write(self: *RingBuffer, data: []const f32) u32 {
        const available = self.capacity - (self.write_pos - self.read_pos);
        const to_write = @min(data.len, available);
        // 环形写入，处理 wrap-around
        const first_part = @min(to_write, self.capacity - (self.write_pos % self.capacity));
        @memcpy(self.buffer[self.write_pos % self.capacity..][0..first_part], data[0..first_part]);
        if (first_part < to_write) {
            @memcpy(self.buffer[0..to_write - first_part], data[first_part..to_write]);
        }
        self.write_pos += to_write;
        return to_write;
    }
};
```

**Cache Line 隔离**：`read_pos` 和 `write_pos` 分别对齐到 64 字节边界，避免 False Sharing 导致的 Cache Line 乒乓效应。在 x86-64 架构上，Cache Line 大小为 64 字节，两个指针位于同一 Cache Line 会导致无谓的缓存同步开销。

#### 3.2.3 SIMD 优化策略

音频 DSP 是 SIMD 优化的理想场景——大量独立的样本级运算（乘法、加法、乘加）天然适合向量化。核心优化策略：

- **4-wide 向量化（SSE/NEON）**：一次处理 4 个 f32 样本，Biquad 滤波器可达到 4x 理论加速。
- **8-wide 向量化（AVX2）**：一次处理 8 个 f32 样本，FFT 蝶形运算的吞吐量翻倍。
- **16-wide 向量化（AVX-512）**：一次处理 16 个 f32 样本，适用于大规模混音和 FDN 矩阵运算。

```zig
// Biquad Direct Form II Transposed - SIMD 优化伪代码
pub fn processBiquadSimd(b: *BiquadState, input: []const f32, output: []f32) void {
    // 使用 4-wide SIMD 处理
    const vec_len = input.len / 4;
    for (0..vec_len) |i| {
        const x = load4(input[i*4..]);           // 加载 4 个输入样本
        const y = b.b0 * x + b.z1;                // FMA: 融合乘加
        store4(output[i*4..], y);                  // 存储 4 个输出样本
        b.z1 = b.b1 * x - b.a1 * y + b.z2;       // 更新延迟线
        b.z2 = b.b2 * x - b.a2 * y;               // 更新延迟线
    }
}
```

### 3.3 内存模型

#### 3.3.1 环形缓冲设计

环形缓冲区是音频流式加载的核心数据结构。设计参数：

- **缓冲大小**：256 样本（约 5.3ms @48kHz），平衡延迟与稳定性。此值来源于 AAA 游戏的工程实践：小于 128 样本会增加调度开销和丢帧风险，大于 512 样本会增加可感知的音频延迟 [8]。
- **双缓冲策略**：流式加载使用双缓冲（Double Buffering），一个缓冲用于播放，另一个用于异步填充。切换时机由播放位置决定，确保无缝衔接。

#### 3.3.2 预分配池与 Cache 友好布局

音频引擎的所有动态数据结构（Voice、EffectProcessor、Bus Node）均从预分配的内存池中分配。内存池在引擎初始化时一次性分配，运行时仅执行指针移动（Bump Allocation），零碎片产生。

**Cache 友好布局**：Voice 对象按访问频率分层存储：

```
[热区] VoiceState (64B)     — 每帧访问：位置、音量、状态标志
[温区] VoiceParams (128B)   — 参数变化时访问：衰减曲线、效果器参数
[冷区] VoiceMeta (256B)     — 仅初始化/销毁时访问：名称、ID、资产引用
```

热区数据紧凑排列，确保在 128 样本（约 2.7ms）的处理窗口内，所有活跃 Voice 的热区数据都能驻留在 L1 Cache 中。

## 4. 性能预算与优化策略

### 4.1 CPU 与内存预算

基于 AAA 游戏的实际生产数据，音频引擎的运行时预算如下 [12]：

| 预算项 | 目标值 | 峰值容忍 | 说明 |
|:---|:---|:---|:---|
| **CPU 帧时间** | <1.5ms (30fps) | <2.5ms | 占总帧预算的 5-8% |
| **音频线程** | 独占 1 核心 | 1.5 核心 | 高优先级实时线程 |
| **内存占用** | <50MB | <80MB | 含所有预加载资产 |
| **活跃 Voice** | 64-128 | 256 | 超出触发 Voice Stealing |
| **DSP 效果器** | 16 个/帧 | 32 个/帧 | 超出使用 Virtual Voice |

**The Last of Us Part II** 的音频 Foley 系统占用了超过 50% 的关卡内存预算，这一数据警示我们：音频资产的内存占用不可忽视，必须在资产管线阶段进行压缩和去重 [12]。

### 4.2 SIMD 批量处理

SIMD 优化贯穿整个 DSP 管线。关键优化点：

- **Biquad 滤波器**：4-wide SIMD 处理，4 个 Biquad 并行计算，适用于多频段 EQ。
- **混音累加**：8-wide SIMD 的融合乘加（FMA）指令，将 `accumulate + multiply` 合并为单指令。
- **FFT 蝶形运算**：利用 AVX2 的 8-wide 向量化，加速 FFT 的复数乘法和加法。

### 4.3 硬件加速接口预留

L1 层预留了硬件加速接口，用于对接现代主机的专用音频硅片：

- **PS5 Tempest Engine**：通过 GPU Compute Queue 提交 HRTF 卷积任务，支持数百个音源的并行处理。接口设计为 `submitHRTFBatch(sources: []Source, hrtf: HRTF)` 的异步模型 [12]。
- **Xbox Project Acoustics**：通过专用 API 提交声学射线追踪任务，硬件加速的衍射和遮挡计算。接口设计为 `queryOcclusion(source: Source, listener: Listener) -> OcclusionResult` 的查询模型 [12]。

这些接口在 PC 平台上通过软件回退实现，确保跨平台一致性。

## 5. 数据驱动与生产管线

### 5.1 音频资产元数据结构

音频资产的元数据在编译期生成，运行时不可修改。元数据结构设计遵循以下原则：

- **单一数据源**：所有音频参数（音量、衰减、循环点）在资产源文件中定义，编译后嵌入 Bank。
- **平台差异化**：同一逻辑音效在不同平台上可编译为不同编码格式（PC: Vorbis, Mobile: ADPCM, Console: PCM）。
- **版本兼容**：Bank 格式包含版本号，支持向前兼容和运行时校验。

### 5.2 热更新支持

基于用户现有的 RenderScript DSL 和通用 VM 架构，音频引擎支持函数级热更新：

- **效果器参数热更新**：通过 `setParameter` 接口实时更新效果器参数，无需重启音频线程。
- **效果器代码热更新**：通过 VM 的 AOT+Interpreter 混合模式，替换效果器的字节码实现，支持 Zig/C 语言的热更新。
- **资产热更新**：通过 Bank 的增量编译和原子替换，在不中断播放的前提下更新音频资产。

### 5.3 编辑器集成接口

编辑器通过 C API 与音频引擎通信，支持以下工作流：

1. **实时预览**：在编辑器中触发音效，引擎返回实时音频流。
2. **参数调试**：可视化衰减曲线、混响参数，实时调整并监听效果。
3. **Bank 编译**：一键编译所有音频资产为平台优化的 Bank 文件。
4. **性能分析**：导出运行时统计（Voice 数量、CPU 时间、内存占用），辅助性能优化。

```zig
// L4: Biquad 滤波器 - Direct Form II Transposed
// ============================================================
pub const BiquadCoeffs = struct {
    b0: f32, b1: f32, b2: f32,
    a1: f32, a2: f32,
};

pub const BiquadState = struct {
    coeffs: BiquadCoeffs,
    z1: f32,  // 延迟线 1
    z2: f32,  // 延迟线 2

    pub fn process(self: *BiquadState, input: []const f32, output: []f32) void {
        for (input, 0..) |x, i| {
            const y = self.coeffs.b0 * x + self.z1;
            output[i] = y;
            self.z1 = self.coeffs.b1 * x - self.coeffs.a1 * y + self.z2;
            self.z2 = self.coeffs.b2 * x - self.coeffs.a2 * y;
        }
    }

    pub fn reset(self: *BiquadState) void {
        self.z1 = 0.0;
        self.z2 = 0.0;
    }
};

    requested,

pub const VoiceManager = struct {
    voices: []Voice,            // 预分配对象池
    max_active: u32,            // 最大活跃语音数
    active_count: u32,

    pub fn stealIfNeeded(self: *VoiceManager) void {
        if (self.active_count <= self.max_active) return;

        // 按优先级排序（低优先级 = 高数值）
        std.sort.insertion(Voice, self.voices, {}, Voice.lessThanByPriority);

        var stolen_count: u32 = 0;
        const need_to_steal = self.active_count - self.max_active;

        for (self.voices) |*voice| {
            if (voice.state != .active) continue;
            if (stolen_count >= need_to_steal) break;

            // 执行 Voice Stealing
            voice.state = .stolen;
            voice.ring_buffer.reset();
            stolen_count += 1;
        }

        self.active_count -= stolen_count;
    }

    fn lessThanByPriority(_: void, a: Voice, b: Voice) bool {
        // 优先级数值越大 = 越低
        if (a.priority != b.priority) return a.priority > b.priority;
        // 同等优先级：比较 RMS 电平
        if (a.rms_level != b.rms_level) return a.rms_level < b.rms_level;
        // 同等 RMS：比较距离衰减
        return a.distance_attenuation < b.distance_attenuation;
    }
};

// L6: FDN 混响 - 16 路延迟线 + Householder 反馈矩阵
// ============================================================
pub const FdnReverb = struct {
    delay_lines: [16]DelayLine,
    feedback_matrix: [16][16]f32,
    pre_delay: DelayLine,
    rt60: f32,
    diffusion: f32,

    pub fn init(rt60: f32, pre_delay_ms: f32) FdnReverb {
        var self = FdnReverb{
            .delay_lines = undefined,
            .feedback_matrix = undefined,
            .pre_delay = DelayLine.init(@floatToInt(u32, pre_delay_ms * 48.0)),
            .rt60 = rt60,
            .diffusion = 0.7,
        };

        // 初始化 Householder 反馈矩阵
        const h: f32 = 2.0 / 16.0;  // N=16
        for (0..16) |i| {
            for (0..16) |j| {
                self.feedback_matrix[i][j] = if (i == j) 1.0 - h else -h;
            }
        }

        // 计算延迟线长度（质数分布，避免共振）
        const prime_lengths = [_]u32{ 1493, 1559, 1613, 1693, 1741, 1783, 1861, 1913,
                                      1993, 2053, 2113, 2179, 2237, 2293, 2357, 2417 };
        for (0..16) |i| {
            self.delay_lines[i] = DelayLine.init(prime_lengths[i]);
        }

        return self;
    }

    pub fn process(self: *FdnReverb, input: []const f32, output: []f32) void {
        // 1. 预延迟
        const delayed = self.pre_delay.process(input);

        // 2. 馈入延迟线
        for (0..16) |i| {
            self.delay_lines[i].write(delayed[i] * (1.0 / 4.0));
        }

        // 3. 反馈矩阵混合
        var feedback: [16]f32 = undefined;
        for (0..16) |i| {
            var sum: f32 = 0.0;
            for (0..16) |j| {
                sum += self.feedback_matrix[i][j] * self.delay_lines[j].read();
            }
            feedback[i] = sum * self.diffusion;
        }

        // 4. 输出混合
        for (output, 0..) |*out, i| {
            out.* = feedback[i % 16];
        }
    }
};

// L7: HRTF 双线性插值

pub const HrtfDataset = struct {
    hrir_data: []f32,           // 扁平化的 HRIR 数据集
    azimuth_count: u32,
    elevation_count: u32,
    tap_count: u32,             // 典型值：128 或 256

    pub fn getHrir(self: *const HrtfDataset, azimuth: f32, elevation: f32) []const f32 {
        // 双线性插值：找到最近的 4 个 HRIR 采样点
        const az_idx = self.findAzimuthIndex(azimuth);
        const el_idx = self.findElevationIndex(elevation);

        const az_frac = azimuth - self.azimuths[az_idx];
        const el_frac = elevation - self.elevations[el_idx];

        // 插值权重
        const w00 = (1.0 - az_frac) * (1.0 - el_frac);
        const w01 = (1.0 - az_frac) * el_frac;
        const w10 = az_frac * (1.0 - el_frac);
        const w11 = az_frac * el_frac;

        // 从 4 个 HRIR 混合
        const hrir00 = self.getHrirAt(az_idx, el_idx);
        const hrir01 = self.getHrirAt(az_idx, el_idx + 1);
        const hrir10 = self.getHrirAt(az_idx + 1, el_idx);
        const hrir11 = self.getHrirAt(az_idx + 1, el_idx + 1);

        // 加权混合
        var interpolated: [256]f32 = undefined;
        for (0..self.tap_count) |i| {
            interpolated[i] = w00 * hrir00[i] + w01 * hrir01[i] +
                             w10 * hrir10[i] + w11 * hrir11[i];
        }

        return interpolated[0..self.tap_count];
    }
};

pub const StreamingVoice = struct {
    voice: Voice,
    front_buffer: RingBuffer,   // 播放缓冲
    back_buffer: RingBuffer,    // 填充缓冲
    io_thread: std.Thread,
    is_streaming: bool,

    pub fn startStreaming(self: *StreamingVoice, file_path: []const u8) !void {
        self.is_streaming = true;
        // 在 I/O 线程中异步填充 back_buffer
        self.io_thread = try std.Thread.spawn(.{}, fillBuffer, .{ self, file_path });
    }

    fn fillBuffer(self: *StreamingVoice, file_path: []const u8) void {
        const file = std.fs.cwd().openFile(file_path, .{}) catch return;
        defer file.close();

        var buf: [4096]u8 = undefined;
        while (self.is_streaming) {
            const bytes_read = file.read(buf[0..]) catch break;
            if (bytes_read == 0) break;  // EOF

            // 解码并写入 back_buffer
            const samples = decodeToFloat(buf[0..bytes_read]);
            _ = self.back_buffer.write(samples);

            // 等待 front_buffer 消费完毕，交换缓冲
            while (self.front_buffer.write_pos - self.front_buffer.read_pos > self.front_buffer.capacity / 2) {
                std.time.sleep(1 * std.time.ns_per_ms);
            }
        }
    }
};
```

### 6.2 混音循环完整实现

```zig
// ============================================================
// 主混音循环 - 每 256 样本（~5.3ms）执行一次
// ============================================================
pub fn audioThreadMain(mixer: *AudioMixer, device: *AudioDevice) void {
    // 设置实时线程优先级
    std.os.sched_setscheduler(0, .fifo, .{ .priority = 99 }) catch {};

    var output_buffer: [256]f32 align(64) = undefined;  // Cache Line 对齐

    while (true) {
        // 1. 从设备获取输出缓冲区
        const output = device.getBuffer() catch continue;

        // 2. 执行混音循环
        mixCycle(mixer, output_buffer[0..]);

        // 3. 将混音结果写入设备缓冲区
        @memcpy(output, output_buffer[0..]);

        // 4. 提交音频帧
        device.commitBuffer(256) catch {
            std.debug.print("Audio dropout detected!\n", .{});
        };
    }
}

fn mixCycle(mixer: *AudioMixer, output: []f32) void {
    // 清零输出缓冲区
    @memset(output, 0.0);

    // Voice Stealing 判定
    mixer.voice_manager.stealIfNeeded();

    // 遍历所有活跃 Voice
    for (mixer.voice_manager.voices) |*voice| {
        if (voice.state != .active) continue;

        // 从环形缓冲区读取数据
        const input = voice.ring_buffer.read(256);
        if (input.len == 0) {
            voice.state = .releasing;
            continue;
        }

        // 计算增益：包络 * 距离衰减 * Bus 音量
        const gain = voice.envelope.tick() *
                    voice.distance_attenuation *
                    mixer.bus_graph.getBusVolume(voice.bus_id);

        // 累加到目标 Bus
        const bus = &mixer.bus_graph.buses[voice.bus_id];
        for (output, input, 0..) |*out, in, i| {
            out.* += in * gain;
        }
    }

    // 遍历 Bus 图，执行效果器链
    mixer.bus_graph.processEffects(output);

    // 最终限制器（防止削波）
    applyLimiter(output);
}

fn applyLimiter(buffer: []f32) void {
    const threshold: f32 = -1.0;  // dBFS
    const release_ms: f32 = 50.0;
    var peak: f32 = 0.0;

    for (buffer) |sample| {
        const abs_sample = @abs(sample);
        peak = @max(peak, abs_sample);
    }

    if (peak > 1.0) {
        const gain_reduction = 1.0 / peak;
        for (buffer) |*sample| {
            sample.* *= gain_reduction;
        }
    }
}
```
